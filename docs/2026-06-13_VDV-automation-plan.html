VDV Automation Plan3 / 12
Architecture decision · ADR-002

Two modules, vendor-family adapters, one registry

① Subject verification

Always from the county assessment source of record, per-vendor adapter, cross-checked with public portals.

② Comp pull

MLS-first. County sales-GIS used only where it exists, as an accelerator.

  • Adapters by vendor family, not per county — ~8 families cover all ~95 VA jurisdictions.
  • MLS is the backbone — only the MLS has DOM, concessions, condition, list-vs-sold, prior-sale chain.
  • No per-county skills — a new county is a registry row, not new code.

VDV Automation Plan6 / 12
Recurring pain · now a playbook

Resolving a subject with no Tax ID

  • Normalize + expand the address — street-type / directional / unit variants (county ≠ MLS ≠ postal).
  • Assessment by address → geocode + point-in-parcel to get the GPIN even when the address attribute differs.
  • New-construction / builder-parcel check — units may sit under an undivided builder parcel (SF=0) for 1–2 yrs.
  • Use the nearest assessed neighbor unit as the GLA/spec proxy → start the DM file by entering the subject manually, then import comps.
  • Cross-check MLS (all 3) + portals for the Tax ID and the alternate street name; record the alias mapping.
  • Cache every resolution in SQLite — the streamlining win.

Plain-English · for Alan

How "ACI web auto-fill" actually works

1 · Browser fills the form

New ACI is a website, so the agent types each value into the right box like a person — GLA, year built, every comp price. Nothing invented; all of it comes from the structured record.

2 · A swappable field-map

A small editable table says which value goes in which ACI box. When ACI changes its layout, you edit this map — a data change — not the code, so updates don't break it.

3 · You review & submit

The agent fills, then stops. It never clicks Save / Submit. You check the form and press the button — exactly like DataMaster's "Review & Send."

field-map  (our data  →  ACI's form field)
  subject.gla          →  #GLA
  subject.year_built   →  #YearBuilt
  comp[1].sale_price   →  #comp1_SalePrice
  comp[1].gla          →  #comp1_GLA

Net: the machine does the tedious typing; the licensed appraiser stays the final gate. (Can't build until the new ACI is live and we can see its real fields — HTML copy-paste is the bridge until then.)

The business case · scope with Alan

Desktop appraisals — the volume play

Why they fit automation

No interior/exterior inspection — the whole report is built from data (MLS, public records, prior listings, floor plan). That data-assembly is exactly what this pipeline already does.

Why it pays for itself

Far lighter than a full 1004/2055, so we can take on more of them. The added throughput funds ops — MLS dues, Claude subscriptions — then turns to margin. A self-funding flywheel.

  • Same engine, lighter form — subject + comps + market data → the desktop form; reuses everything in Phases 1–5.
  • Appraiser still certifies — automation assembles; the licensed appraiser judges, signs, and stays the gate (USPAP unchanged).
  • Confirm scope with Alan — which desktop products / forms, which AMCs, and the eligibility rules.
